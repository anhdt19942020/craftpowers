# Generate Project Context

## Overview

Scan the codebase and interview the user to produce `project-context.md` — a single document that serves as shared context ("constitution") for all agents working on this project. This document answers: what is this project, who is it for, what are the hard constraints, and what decisions have already been made.

**Announce at start:** "I'm using the generate-project-context skill to create the project context document."

**Output:** `docs/mankit/project-context.md`
- Do NOT commit — this is a living working document
- If the file already exists, update it rather than overwrite (confirm changes with user)

## Why This Exists

Agents dispatched per-task (triệu-vân, bàng-thống, etc.) start with zero project knowledge. Without shared context, each agent:
- Re-discovers constraints the team already decided on
- Makes assumptions that contradict project philosophy
- Produces code that works but doesn't fit

`project-context.md` eliminates this by giving every agent the same baseline understanding before they touch code.

## Process

### Step 1: Scan Existing Context

Gather what already exists — don't ask the user things the codebase already answers:

1. **Project identity:** `package.json`, `Cargo.toml`, `pyproject.toml`, `README.md` — name, description, language, dependencies
2. **Existing docs:** `docs/**/*.md`, `CLAUDE.md`, `AGENTS.md` — conventions already documented
3. **Architecture signals:** directory structure, entry points, test patterns
4. **Git history:** `git log --oneline -20` — recent direction and velocity
5. **CI/CD:** `.github/workflows/`, `Makefile`, `docker-compose.yml` — how it builds and deploys

Present findings to user: "Here's what I gathered from the codebase. I'll ask a few questions to fill gaps."

### Step 2: Interview (Fill Gaps Only)

Ask only what the codebase doesn't answer. One question at a time. Skip any question the scan already answered.

**Identity & Purpose:**
- What problem does this solve? Who uses it?
- What's the one thing this project must never compromise on? (performance? correctness? simplicity? developer experience?)

**Constraints:**
- Are there hard technical constraints? (must run on X, can't use Y, must support Z)
- Are there business constraints? (compliance, timeline, team size)

**Decisions Already Made:**
- What architectural decisions are locked in? (monolith vs microservices, DB choice, auth approach)
- What was tried and rejected? (so agents don't re-propose it)

**Non-Goals:**
- What does this project explicitly NOT do? (prevents scope creep by agents)

### Step 3: Generate Document

Write `docs/mankit/project-context.md` using this structure:

```markdown
# Project Context

> This document is shared context for all agents working on this project.
> Generated by `generate-project-context` skill. Last updated: YYYY-MM-DD.

## Identity

**Name:** [project name]
**Purpose:** [one sentence — what it does and for whom]
**Language/Stack:** [primary language, framework, key dependencies]
**Repo:** [monorepo? single package? workspace?]

## Core Principle

[The one thing this project must never compromise on. One sentence.]

## Architecture

[2-5 sentences describing the high-level structure. Entry points, data flow, key boundaries.]

## Hard Constraints

- [constraint 1 — e.g., "zero runtime dependencies"]
- [constraint 2 — e.g., "must work offline"]
- [constraint 3]

## Decisions Already Made

| Decision | Chosen | Rejected Alternatives | Why |
|----------|--------|----------------------|-----|
| [topic] | [choice] | [what was considered] | [reason] |

## Non-Goals

- [thing this project explicitly does NOT do]
- [another non-goal]

## Conventions

[Key patterns agents must follow. Reference CLAUDE.md if it exists rather than duplicating.]

## Testing Strategy

[How this project tests: unit? integration? e2e? what framework? what's the run command?]

## Deployment

[How it ships: CI/CD? manual? what environments exist?]
```

### Step 4: User Review

Present the document and ask user to review. Fix any corrections. Only after approval, write the file.

## Consumers

Once `project-context.md` exists, these skills read it automatically:

- **writing-plans** — includes it in plan header context so implementers have project awareness
- **subagent-driven-development** — passes it to each dispatched agent
- **brainstorming** — uses it to ground design proposals in project reality

## When to Re-run

- After major architectural changes
- When onboarding the project to mankit for the first time
- When agents keep making the same wrong assumptions (symptom of missing/stale context)
